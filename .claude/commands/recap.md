# 需求重组与提炼

现在进入**需求澄清前的一次重组**环节。你的任务不是开始实现，而是先把用户在本对话中**已经表达过的内容**读回来，重组、提炼成清晰准确的需求陈述，确认你真正理解了用户的意思。

## 你需要做的

1. **回溯本对话**：通读用户在本会话中此前的所有发言（包括本命令后附的 `$ARGUMENTS` 补充说明），不要遗漏、不要臆造用户没说过的内容。

2. **重组提炼**，用结构化的方式输出，至少包含：
   - **用户想要达成什么** —— 用一句话概括核心目标。
   - **关键需求点** —— 拆解为可逐条确认的要点列表。
   - **涉及的边界/范围** —— 用户明确提到要做的、以及明确不做的。
   - **我的理解 vs. 用户的原话** —— 对于模糊或可多重理解的地方，写出"我理解为 X，因为你说了 '…'"，把推断锚定到原文。

3. **列出待澄清的问题**：把所有歧义、缺口、需要用户决策的点单独列出（编号、具体、可回答），不要混进上面的陈述里。问题数量克制，只列真正影响后续方向的。

4. **显式停住**：在结尾明确表示"在确认以上理解和待澄清问题之前，我不会进入 plan 或修改代码"。**不要**调用 ExitPlanMode、不要写代码、不要进入实现。

## 约束

- 严格区分"用户明确说过的"与"我推断的"，后者必须标注为推断。
- 不要在此时提出实现方案、技术选型或文件改动建议——那是确认后的下一步。
- 如果用户在本对话中几乎没说过实质内容，直接说明信息不足，并问用户想做什么，不要编造需求。
